# 朝のニュースおすすめ（プッシュ）機能 — 構想メモ

## ゴール

- **興味領域**に沿ったニュースを集約し、**有用性をランク付け**したうえで、朝のひと声として  
  **「ボス、今日の◯◯（分野）のニュース、ご覧になりますか？」** と提案する。
- ボスが **Yes / あとで / 興味ない** と選べる。選んだ分だけ要約・リンクを返す（チャット内で完結させる）。

## 体験イメージ（朝）

1. **既存の朝フロー**（`/morning` や起動時）の末尾に 1〜2 行追加するイメージ。  
   - 例: *「本日は生成AI広告と日本のスタートアップ資金調達で、押さえどころが2件あります。ご覧になりますか？」*
2. ボス: **「見る」「AIだけ」「いらない」** など短文でフィルタ。
3. 影: **ランク順に最大3件**、タイトル・1行要約・ソース名・URL（任意でNotionメモに保存）。

※「プッシュ」は **PWA / メール / Slack** まで広げられるが、**フェーズ1はアプリ内＋朝メッセージ**が現実的。

## データの流れ

```
[ニュースソース] → [取得・正規化] → [重複排除] → [スコアリング] → [朝用1ブロック生成] → KAGE表示
                      ↑
            Profile / 会話から「興味タグ」を同期（任意）
```

## ニュースソース（案）

| 方式 | 長所 | 短所 |
|------|------|------|
| **RSS/Atom**（Tech系メディア、官公庁、業界団体） | 実装簡単・コスト低 | サイトごとに質差 |
| **Google News RSS**（キーワード） | 網羅 | ノイズ多め・利用規約要確認 |
| **有料API**（NewsAPI等） | 整形データ | 課金・キー管理 |
| **X/Reddit等** | 速い | ノイズ・規約・認証が重い |

**推奨**: フェーズ1は **許諾の取りやすいRSSをホワイトリスト化** + ボスが **Notion Profile または環境変数**で「フォローするフィードURL」を持てるようにする。

## ランク付け（有用性スコア）

ルール＋LLMのハイブリッドが現実的。

1. **ハードフィルタ**: 重複、明らかなスパム、古すぎる記事（例: 36h超）を落とす。  
2. **ルール加点**: ボスの興味タグ一致、タイトルに固有名詞（勤務先・講義テーマ等）、カテゴリ一致。  
3. **LLMスコア**（バッチ・安いモデル）:  
   - 要約可能性、新規性、**CA/講義/制作業務への有用度** を 1〜5 で付与。  
   - プロンプトに **BOSS_PROFILE** を渡す（既存の秘書プロンプトと整合）。  
4. **最終スコア** = ルール + LLM（重み付け）。**上位N件だけ**を朝文案に渡す。

## パーソナライズ（興味の学習）

- **既存**: Profile DB、会話の auto_learn。  
- **追加案**:  
  - `興味ニュース` 用の **タグ一覧**（Notion Profile または専用DB）  
  - 「このジャンルはもういい」→ **ミュートタグ** を保存  
  - クリック/「見た」ログが取れれば **学習ループ** に使える（後回しで可）。

## 「朝プッシュ」の技術的な意味

| レベル | 内容 |
|--------|------|
| L1 | 朝ブリーフィングAPIのレスポンスに **news_teaser** を足す（KAGEを開いたときに見える） |
| L2 | **PWA + Web Push**（Service Worker、VAPID、購読管理） |
| L3 | **メール / Slack / LINE** への同文面送信（Webhook・SendGrid等） |

**まず L1** で価値検証 → 需要があれば L2/L3。

## サーバー側の追加コンポーネント（案）

- **ジョブ**: 1日1〜2回（例: 5:30 / 6:00 JST）に `fetch → score → cache`。  
  - Railway Cron / 外部 cron が `GET /internal/news/digest` を叩く（**共有シークレット必須**）。  
- **キャッシュ**: Redis またはファイル/SQLite（小規模なら **当日分JSON** で十分）。  
- **エンドポイント**:  
  - `GET /morning` 拡張: `news_digest` をマージ  
  - または `GET /news/teaser` をフロントが起動時に取得  

## UX・コピー

- 押し付けない: **「ご覧になりますか？」** で止める。  
- **1ジャンル1問**にすると負担が少ない（「今日のAIニュースは？」）。  
- バグ・整理機能と同様、**Notionに「読んだメモ」**を残すオプションがあると後から検索しやすい。

## リスク・注意

- **著作権・要約**: 長文の無断転載は避け、**リンク＋短い要約**に留める。  
- **幻覚**: スコアリングは「元記事タイトル＋リード」だけ渡す。  
- **コスト**: LLMは **記事あたりトークン制限**、1日の上限件数。  
- **政治・炎上**: ホワイトリストRSS＋カテゴリ制限で初期は守る。

## フェーズ分割（おすすめ）

1. **P0**: RSSホワイトリスト1〜3本 + 取得 + 重複排除 + **朝ブリーフに1段落挿入**（ランクは単純ルールのみ）。  
2. **P1**: Profile連動の興味タグ + LLMスコア + 「見る/スキップ」会話フロー。  
3. **P2**: クリックログ + スコア調整 + PWAプッシュ検討。

## 実装状況（P0）

| 項目 | 内容 |
|------|------|
| モジュール | `news_digest.py` — RSS取得（**生データキャッシュ**）・年齢フィルタ・**brain＋環境の重み付き**スコア（毎回再計算）・ミュート |
| 環境変数 | `KAGE_NEWS_RSS_FEEDS`, `KAGE_NEWS_KEYWORDS`, `KAGE_NEWS_MUTE_KEYWORDS`, `KAGE_NEWS_ENV_WEIGHT`, `KAGE_NEWS_MINING_PROFILE`, `KAGE_NEWS_MAX_AGE_HOURS`, `KAGE_NEWS_CACHE_SEC`, `KAGE_NEWS_TOP_N`, `KAGE_NEWS_CRON_SECRET` |
| Notion | Profile は **last_edited 降順**で取得（最近の関心ルールが先）。`ニュース関心` / `ニュース除外` / カテゴリ `ニュース` を解釈 |
| API | `GET /morning` が `news`（`interest`, `keywords_env`, `rss_cached_at` 含む）を返す。`GET /news/digest` は既定 **brain なし**（env のみ）。Cron は RSS 更新用 |
| 朝UI | ブリーフ内に折りたたみ「本日のRSSリンク」 |
| 運用メモ | **`NEWS_INTEREST_OPS.md`** ・ **`SECRETARY_CRAFT.md`**（秘書の引き出し方） |
| 感想ループ | 朝ブリーフ＋ニュース後に **感想プロンプト** → チャット1往復 → Notion **`[ニュースFB]`** メモ → 次回 `news_digest` の重みに反映 |
| 依存 | `feedparser` |

**次の一手（P1）**: Gemini でタイトル+要約だけ採点、チャット「ニュース見る」意図で詳細列挙。
