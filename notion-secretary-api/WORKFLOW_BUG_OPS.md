# バグ報告の運用（KAGE × Notion × 開発）

「バグを溜める」だけで終わらせず、**直すまでの流れ**を決めておくためのメモです。

## ステータス（Notion・デバッグログ DB）

| ステータス | 意味 |
|------------|------|
| **未対応** | いま手を付けていない（受信箱） |
| **対応中** | 原因調査 or 修正中・デプロイ待ち |
| **完了** | 本番に反映済み、または「仕様で却下」などクローズした |

KAGE ヘッダの **リストアイコン** → 各項目の **未対応 / 対応中 / 完了** ボタンで更新できます（Notion と同期）。

## おすすめフロー

1. **発見したらすぐ**  
   KAGE で `バグ: 〇〇` または 🐛 ボタン。会話コンテキストが一緒に残る。

2. **受信箱を見る**  
   リストを開き、フィルタ **「未対応」** で優先順を決める。

3. **手を付け始めたら**  
   その項目を **「対応中」** にする（自分へのコミットにもなる）。

4. **修正**  
   - 会話コンテキストを **コピーログ** で Cursor に貼る、または Notion を開く。  
   - コード修正 → `git push` → Railway などへデプロイ。

5. **本番で確認できたら**  
   KAGE または Notion で **「完了」**。

6. **週1でも可**  
   「未対応」の件数だけ見て、潰す or 優先度を下げる判断。

## API（自動化・メモ用）

- `GET /debug/recent?status=未対応` … 受信箱だけ  
- `POST /debug/status` … `{"page_id":"...", "status":"対応中"}`  

## うまく回すコツ

- **再現手順**が足りないときは、同じスレッドでもう一言 `バグ: 補足で〇〇` と送れば別行として残せる。  
- **仕様かバグか曖昧**なものは Notion のメモ欄に一言足す。  
- 完了は **デプロイ後** に付けると、あとから見ても安心。
